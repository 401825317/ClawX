# 反模式与常见错误用法

> 本文件列出 **不该怎么做** 的用法。遵循这些规则可避免 90% 的"文件打不开/乱码/进程卡死"问题。
> 遇到报错先查 `troubleshooting.md`，再回看这里。

---

## 🔴 新手高频踩坑（必读）

以下是最常见的 **操作层面错误**，不涉及技术细节，但 80% 的新手反馈问题都出自这里。

### 坑 1：「转换后原文件变了？」——不会的

**误解：** 调完 `office_to_pdf('报告.docx', '报告.pdf', 'docx')` 后，以为原来的 `报告.docx` 内容会被修改或删除。

**事实：** 所有转换都是**读取源文件 → 生成新文件**，原文件**永远不会被修改**。PDF 是新产生的，`报告.docx` 完好无损。

---

### 坑 2：一次性转 50 个文件 → 内存/进程爆炸

**错误做法：**
```python
# 写个循环一口气转 50 个 DOCX→PDF
for f in glob('*.docx'):
    office_to_pdf(f, f.replace('.docx','.pdf'), 'docx')
```

**后果：** WPS 进程反复启动退出，第十几个开始卡住；或内存持续上涨直到 OOM。

**正确做法：**
- 每批 **≤10 个**，批间 `time.sleep(3)` 让进程完全释放
- 每个文件独立 `try-except`，一个失败不影响其余
- 批量场景参考下方「推荐做法速查」表的批量示例

---

### 坑 3：用 `create_xlsx()` 去编辑已有文件

**误解：** 以为 `create_xlsx('已有报表.xlsx', new_data)` 能往现有 Excel 里追加数据。

**事实：** `create_xlsx()` **永远创建新文件**，如果路径已存在会**直接覆盖**。它不是"编辑器"。

**正确做法：**
- 要**读取**已有 XLSX → 用 `read_xlsx()`
- 要**修改**已有 XLSX → 读出来 → 在内存里改 → 用 `create_xlsx()` **写到新路径** → 确认 OK 后替换原文件
- 绝不要让输出路径 = 输入路径

---

### 坑 4：路径里有空格或中文引号

**错误做法：**
```python
office_to_pdf('D:\\我的文档\\2026年Q1 报告.docx', 'out.pdf', 'docx')
#                              ↑ 空格
create_xlsx('C:\\Users\\张三\\"新建文件夹"\\data.xlsx', data)
#                    ↑ 中文引号
```

**后果：** COM 引擎解析路径失败，报找不到文件（但肉眼看着路径明明对的）。

**正确做法：**
- 路径**不要手动拼字符串**，始终用 `os.path.join()`
- 文件名避免空格、括号、`#` `&` `%` 等特殊字符
- 如果无法避免（比如用户给的原始文件名），先 `shutil.copy()` 到一个纯 ASCII 路径再操作

---

### 坑 5：以为 `read_pdf()` 能提取扫描件文字

**误解：** "这个 PDF 我能打开看到字，为什么 `read_pdf()` 返回空的？"

**事实：** 你看到的"字"其实是**图片**（扫描件/拍照件），里面没有文字层。`pdftotext` 只能提取有文字层的 PDF。

**判断方法：**
```bash
# 先测试有没有文字层
pdftotext your.pdf - | wc -c
# 输出 0 = 纯图片 PDF，需要 OCR（本工具不做 OCR）
# 输出 >0 = 有文字层，可以正常提取
```

---

### 坑 6：XLSX 公式引用了不存在的单元格

**错误做法：**
```python
data = [
    ['科目', '数学', '英语', '总分'],
    ['张三', 85, 92, {'formula': 'SUM(B2:D2)', 'style': 2}],  # D2 不存在！只有 B2 C2
]
```

**后果：** Excel/WPS 打开后公式显示 `#REF!` 错误。

**正确做法：** 公式里的列字母必须和实际数据列一一对应。用 `create_xlsx()` 时注意：
- 第1列=A, 第2列=B, 第3列=C, 第4列=D ……
- `SUM(B2:C2)` = 对第2、3列求和（数学+英语），不要写成 `B2:D2`

## ⚠️ 技术层面的反模式

以下适合有一定 Python 基础的用户阅读。如果你是纯小白，上面 6 个坑已经够用了。

### ❌ 反模式 7：对超大文件用全量解析

**错误做法：**
```python
# 50MB 的 XLSX 直接 ElementTree 全量解析 → 内存爆到 250MB+
root = ET.fromstring(z.read('xl/worksheets/sheet1.xml'))
```

**正确做法：**
- 超 2 万行 / 15MB 的 XLSX → 改用 `openpyxl.read_only=True` 流式模式
- 超大 PDF → `pdftotext` 本身已是 C 程序，无上限，直接用
- 本 skill 的 `xlsx_to_csv` / `xlsx_to_json` 内置 15MB 提示，超 100MB 的 COM 转换会被直接拦截

---

### ❌ 反模式 8：COM 转换时文件被占用

**错误做法：**
- WPS/Office 正打开着这个文件 → 调 `office_to_pdf()` 转换它
- 文件设为「只读」权限不足

**后果：** COM 打开失败或写出的 PDF 为空，重试 3 次后仍失败。

**正确做法：**
- 转换前关闭该文件在 WPS/Office 中的所有实例
- 确认输出目录有写入权限和 ≥2×文件大小的磁盘空间
- 本 skill 已内置预清理 + 强制安全网，正常情况不会残留进程

---

### ❌ 反模式 9：创建 XLSX 忽略中文字体

**错误做法：**
```python
# stdlib 手写 styles.xml 用 Calibri / Arial
<font><name val="Calibri"/></font>
```

**后果：** WPS 转 PDF 时中文乱码（显示为"结帐常"之类方块）。

**正确做法：**
- 用 `openpyxl` 创建：`Font(name='微软雅黑')`
- 必须 stdlib 手写时：设 `宋体`/`微软雅黑` + `charset val="134"`
- 本 skill 的 `create_xlsx()` 模板已预设中文字体，直接用即可

---

### ❌ 反模式 10：用 stdlib 手写 PPTX

**错误做法：**
```python
# 试图手写 11 个 slideLayout + thumbnail.jpeg + presProps 等 ~47 个文件
```

**后果：** WPS/Office 直接报"不可读取内容"或空白页。

**正确做法：**
- PPTX 创建**必须用 `python-pptx`**（本 skill `create_pptx()` 已实现）
- WPS 要求 PPTX 最小结构约 47 个内部文件，手写不现实

### ❌ 反模式 11：用西文字体 Calibri 创建含中文的 XLSX

**错误做法：**
```python
create_xlsx('报表.xlsx', data, sheet_name='Report')  # 但模板字体是 Calibri
```

**后果：** 同上，转 PDF 乱码。

**正确做法：** 同反模式 3，字体必须中文化。

---

### ❌ 反模式 12：对同一文件并发读写

**错误做法：**
```python
# 线程 A 写 x.xlsx，线程 B 同时读 x.xlsx
```

**后果：** ZIP 结构损坏，报 `BadZipFile` 或读取到半截数据。

**正确做法：**
- 串行操作，或用临时文件写完后 `os.replace()` 原子替换
- 本 skill 的 `create_xlsx()` 先写临时目录再打包，天然安全

---

### ❌ 反模式 13：指望 pdftotext 提取扫描件文字

**错误做法：**
```bash
pdftotext scan.pdf -   # 扫描件输出为空，以为工具坏了
```

**后果：** 输出 0 字符，误判为 bug。

**正确做法：**
- 扫描件 PDF 是纯图片，无文字层 → 需 OCR（本 skill 不内置 OCR）
- 先验证：`pdftotext file.pdf - | wc -c`，输出 0 即无文字层

---

### ❌ 反模式 14：路径含特殊字符触发 COM 失败

**错误做法：**
```python
office_to_pdf(r'D:\报表#2026&收入%.docx', r'D:\out.pdf', 'docx')
# 路径含 # & % 等特殊字符
```

**后果：** COM 打开失败或路径解析错误。

**正确做法：**
- 转换前把文件移到纯 ASCII/中文路径（无 `#` `&` `%` `空格` 开头等）
- 用绝对路径，避免相对路径歧义

## ✅ 推荐做法速查

| 场景 | 推荐 |
|------|------|
| **批量转 N 个文件** | 每批 ≤10 个，批间 sleep(3)，每个独立 try-except |
| **创建带公式表格** | `create_xlsx()` + `{'formula': 'SUM(...)'}` 单元格 |
| **读取多 Sheet** | `read_xlsx(path, sheet_index='all')` 或 `list_sheets(path)` |
| **超大文件** | `openpyxl.read_only` / 拆分后转换 |
| **中文 PDF 转换** | 确认字体中文 + WPS 已装 |
| **修改已有 Excel** | 读→内存改→写到新路径→确认后替换原文件（绝不覆盖写）|
| **路径有空格/特殊字符** | 先 copy 到纯 ASCII 路径再操作 |
| **PDF 提取为空？** | 先 `pdftotext file.pdf - \| wc -c` 验证是否有文字层 |

## 📌 快速自检清单

开始操作前，花 10 秒过一遍这个清单，能避掉 90% 的坑：

- [ ] 原文件有备份吗？（本工具不修改原文件，但覆盖写新文件时注意路径）
- [ ] 输出路径 ≠ 输入路径？
- [ ] 文件没有被 WPS/Office 打开着？
- [ ] 路径没有空格、括号、`#&%`？
- [ ] 批量操作分批了吗（≤10 个/批）？
- [ ] 公式列字母和实际数据列对得上吗？
